# Ralph Progress Log
## Codebase Patterns
- Keep schema parsing in `application/parsing.py` and semantic business-input checks in `application/validation.py` so Phase 2 and Phase 3 boundaries stay explicit.
- Semantic validation errors should use deterministic `ValidationError` messages prefixed by field path (for example `age: ...`) to keep CLI output and tests stable.
- Evaluate training-session dependency rules against `sum(training_session.zones_minutes.values())`; missing `training_before_meal` is valid only when the sum is `0`.
- Normalize `training_session.zones_minutes` through `normalize_training_zones` before any semantic rule uses zone data; this enforces canonical `1..5` coverage and deterministic key/value validation.
- Keep macro-output hard constraints in `domain/validation.py`; when invariants fail (for example negative macro grams), raise `DomainRuleError` so exit-code mapping remains `DOMAIN`.
- For response meal-structure invariants, validate `list[MealAllocation]` at the domain boundary with `validate_meal_allocation_invariants` using `CANONICAL_MEAL_ORDER` (count, then coverage, then order) for deterministic errors.
- For carb reconciliation checks, call `validate_carb_reconciliation_invariants` and use `CARB_RECONCILIATION_TOLERANCE` (`abs(sum - target) <= 1e-9`) to keep mismatch behavior deterministic across layers.
- Run Phase 3 validation through `application/orchestration.py::validate_meal_plan_flow` to keep one canonical order: `parse_contract` -> `validate_semantic_input` -> domain invariant checks.
- For Phase 3 regression expansion, add cases to `tests/unit/test_validation_matrices.py` and prefer asserting exception class plus `map_exception_to_exit_code(...)` category with minimal message fragments/prefixes.
