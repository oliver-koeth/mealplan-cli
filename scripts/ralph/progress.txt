## Codebase Patterns
- When adding a new `MealPlanRequest` boundary field, update the contract schema and `CONTRACT_UNITS_POLICY`, the Typer flag-to-request mapping in `src/mealplan/cli/main.py`, and both request-boundary test modules (`tests/unit/test_application_contracts.py`, `tests/cli/test_calculate.py`) in the same iteration.
- For bounded numeric CLI inputs, enforce the inclusive range on the Pydantic request contract and cover both direct `MealPlanRequest.model_validate(...)` boundaries and end-to-end CLI exit-code behavior.
- Training-demand formula changes must update both direct domain coverage in `tests/unit/test_domain_services.py` and the application golden snapshots in `tests/golden/application/`, because orchestration flows straight into the canonical response payloads.
- If a training-demand story only pins formula behavior or physiological regressions, prefer adding focused assertions in `tests/unit/test_domain_services.py` and avoid touching orchestration or goldens unless the public payload changes.
- For application stages that only need a subset of request data, prefer a dedicated typed context object in `src/mealplan/application/orchestration.py` so the stage boundary receives only normalized, stage-specific inputs.
- When the public response metric changes but internal fueling stays intact, keep `training_kcal` as the top-level contract field, preserve `training_carbs_g` only inside meal assembly/training-meal logic, and update contracts, renderers, helper tolerances, and success goldens together.
- For CLI output contract renames, pin both the positive field name and the removed field's absence in real subprocess renderer tests, and use ordered key assertions for JSON output so formatter drift is caught outside mocked service tests.
- For contract/documentation updates, verify README and the canonical docs (`docs/REQUIREMENTS.md`, `docs/MODEL.md`, `docs/ARCHITECTURE.md`) against the actual request contracts, orchestration contexts, and domain models so request-only fields do not drift into domain-object descriptions.
