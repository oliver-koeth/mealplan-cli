## Codebase Patterns
- New domain service helpers should be implemented in dedicated modules (for example `domain/energy.py`) and re-exported from `mealplan.domain.__init__` to keep import paths stable across layers.
- Energy formula functions should live in `src/mealplan/domain/energy.py` and follow explicit unit-bearing names (for example `*_kcal_per_day_for`) to keep calculation intent clear across phases.
- Macro formula helpers should live in `src/mealplan/domain/macros.py` with enum-keyed factor mappings and package-level re-exports in `mealplan.domain.__init__`.
- Residual macro impossibilities (for example negative computed fat) should raise `DomainRuleError` with a stable `macro_targets.<field>` prefix so exit-code mapping and tests remain deterministic.
- Typed calculation inputs should use frozen dataclasses in `src/mealplan/domain/model.py` (for example `UserProfile`) and be consumed by domain services instead of passing unstructured dicts.
- Request boundary contracts use strict pydantic primitives (`StrictInt`/`StrictFloat`) for parse-time no-coercion guarantees, while numeric positivity rules belong in `application.validation.validate_semantic_input`.
- Phase-level composed domain APIs should live in `src/mealplan/domain/services.py`, delegate to formula helpers in `energy.py`/`macros.py`, and return typed dataclasses (`MacroTargets`) for application-layer orchestration.
- Floating-point formula regression tests should use `pytest.approx` for multiplicative outputs, and domain error assertions should target stable category prefixes/types rather than full-message snapshots.
- When a phase assumption changes (for example required-vs-default inputs), update both `docs/PLAN.md` and `docs/ARCHITECTURE.md` in the same iteration so backlog and architecture narratives stay consistent.
